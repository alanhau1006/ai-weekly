1.1 企業從「輔助智慧」走向「自主智慧」的三階段演化

Deloitte AI & Insights 負責人 Prakul Sharma 提出了企業智慧成熟度曲線的三個階段，這是目前業界最清晰的框架之一：

📖 Deloitte — From Assisted to Autonomous Intelligence
階段名稱特徵代表應用
第一階段Assisted Intelligence（輔助智慧）AI 和分析幫人解讀資訊，人做最終決策傳統 BI 儀表板、推薦系統
第二階段Augmented Intelligence（增強智慧）機器學習輔助人類決策，人仍在迴圈中GenAI 聊天機器人、Copilot、摘要工具
第三階段Autonomous Intelligence（自主智慧）AI 在設定邊界內自行決策並執行，人設護欄但不驅動每一步Agentic AI 自動採購、自主交易系統
關鍵洞察：目前大多數企業的 GenAI 應用（聊天機器人、文件摘要）停留在第二階段，但真正的經濟價值在第三階段。GenAI 產生一個答案，而自主智慧追求一個結果——它會對目標進行推理、調用工具和資料、根據條件變化調整行動。

「個人副駕駛」的規模化陷阱

TechEx North America 2026 多場會議反覆提及一個現象：「個人 Copilot」陷阱。一個 AI 工具在單一員工桌上表現很好，C-suite 主管自己用了覺得效率提升，全公司都很興奮——但要把這個成功經驗從一個人擴展到整個部門，甚至整個公司，卻處處碰壁。原因是：

📖 TechEx North America 2026 Day 1
  • 個人效率 ≠ 組織效率：單人省下的時間，不等於流程被重構
  • Pilot 用精選資料和冠軍團隊手動跑，掩蓋了缺失的身份控制、過期資料、延遲的合規審查
  • 一旦進入正式部署，這些「治理負債（governance debt）」立刻成為結構性阻礙

1.2 OpenAI 新加坡實驗室與 IMDA Agentic AI 框架

OpenAI 在新加坡設立新區域實驗室，投資 S$3 億（約 US$2.25 億），將招聘 200+ 員工，涵蓋工程、研究、政策等職位。這不只是擴張——它伴隨一個更重要的治理框架：

📖 OpenAI Singapore & IMDA Agentic AI Framework

IMDA 的 Agentic AI 風險分級框架

新加坡資通訊媒體發展局（IMDA）與 OpenAI 合作推出全球首個針對 Agentic AI 的風險分級治理框架，分為四個風險等級：

風險等級定義治理要求
Tier 1 — 極低風險資訊檢索、簡單問答基本透明度要求
Tier 2 — 低風險內部流程自動化、非關鍵任務人類監督 + 活動日誌
Tier 3 — 中風險涉及外部系統互動、財務操作人類在迴圈 + 審批閾值 + 完整審計軌跡
Tier 4 — 高風險自主執行交易、關鍵基礎設施操作即時監控 + 隨時撤銷權限 + 監管機構通報
為什麼重要：這是全球第一個不是「先發展再管制」，而是「隨風險遞增要求遞增治理」的框架。它解決了企業最頭痛的問題——不是不部署 Agentic AI，而是不知道部署到什麼程度需要什麼治理。

1.3 美國政策：川普廢除 AI 行政命令的完整脈絡

川普政府正式廢除了拜登時期的 AI 行政命令（EO 14110），該命令要求大型 AI 模型開發者向政府報告訓練細節和安全測試結果。廢除的背景：

📖 Reuters — Trump repeals Biden AI executive order
  • Musk 和 Zuckerberg 積極遊說：認為報告要求增加合規成本、削弱美國 AI 公司競爭力
  • 核心論述：「對中國競爭力」——如果美國 AI 公司被監管束縛，中國會超越
  • 替代方案：川普政府傾向於由市場自行調節，而非預先監管
  • 實際影響：目前沒有聯邦層面的強制性 AI 安全報告要求，各州自行立法（如加州 SB 1047 的各種變體仍在討論中）

Nvidia H200 中國出口的結構性僵局

一個更具戲劇性的政策故事：川普帶 Jensen Huang 去北京參加川習會，會後對記者說「something could happen」——但什麼都沒發生。

📖 Reuters — Nvidia H200 China export structural deadlock
僵局的真實結構：
  • 美國已授權：約 10 家中國公司（阿里巴巴、騰訊、字節跳動、京東等）獲得最多 75,000 顆 H200 的出口許可，聯想和富士康為授權經銷商
  • 中國不收貨：北京指示中國科技公司將 Nvidia 晶片限用於海外業務，同時支持國產製造
  • 兩個要求互相矛盾：美國要求 H200 只能在中國境內使用；北京要求中國公司只用國產晶片做國內業務
  • 華為趁虛而入：DeepSeek V4 已適配華為 Ascend 晶片進行訓練（不只是推理），這是首次中國前沿模型在訓練層面跑國產晶片
  • Nvidia 中國營收已從 20%+ 跌至約 5%，本季度指引假設中國營收為零
這意味什麼：中國第二大 AI 市場的硬體架構正在被政府指令而非技術基準決定。這不是貿易姿態，是結構性賭注——賭華為效能差距會縮小得夠快。

1.4 歐盟 AI Act 的 Agentic AI 治理挑戰

歐盟 AI Act 在 2026 年進入執行階段，但 Agentic AI 帶來了立法者未預見的挑戰：

📖 AI News — EU AI Act agentic governance challenges

Article 9：持續風險管理

高風險 AI 的風險管理必須是持續的、基於證據的過程，內建於部署的每個階段（開發、準備、生產），並持續審查。

Article 13：可解釋性要求

高風險 AI 系統必須設計成讓部署者能理解其輸出。這意味著：

  • 第三方 AI 系統不能是不透明的程式碼塊
  • 必須提供足夠文件確保安全合法使用
  • 模型選擇和部署方式同時是技術和法規考量

Agentic AI 特有的治理工具

  • Asqav Python SDK：用密碼學簽名每個 agent 的操作，並將所有記錄連結到不可變的 hash chain（類似區塊鏈技術）。如果有人修改或刪除記錄，chain 驗證就會失敗
  • Agentic Asset List（代理資產清單）：企業必須維護所有運行中 agent 的登記冊，每個 agent 需唯一識別，記錄其能力和授予的權限
  • 即時撤銷機制：在高風險場景中，必須能隨時撤銷 agent 的權限

1.5 微軟開源 Agent 治理工具包

微軟發布了開源的 Agent Governance Toolkit，針對 Agentic AI 的運行時安全，這是目前最實用的企業治理工具：

📖 Microsoft Agent Governance Toolkit — GitHub
📖 AI News — Microsoft open sources agent governance toolkit

核心設計：攔截工具調用層

框架在語言模型和企業網路之間放置一個策略執行引擎：

  • 每次 agent 嘗試調用外部工具，toolkit 攔截請求並對照治理規則檢查
  • 如果行為違反策略（例如：只授權讀取庫存數據的 agent 嘗試發送採購訂單），阻擋 API 調用並記錄事件
  • 安全團隊獲得可驗證、可審計的每一個自主決策軌跡
  • 開發者不需要在每個模型的 prompt 中硬編碼安全協議——安全策略與應用邏輯完全解耦

為什麼開源？

  • 如果微軟把這個功能鎖在自家平台，開發團隊會為了趕 deadline 直接繞過它
  • 開源讓安全控制能融入任何技術棧——無論是本地開源模型、Anthropic、還是混合架構
  • 開放標準讓整個資安生態系能在此基礎上建立商業儀表板和事件響應整合

成本控制層面

  • Agentic 系統在推理和執行的連續迴圈中燃燒 API token
  • Toolkit 可設定 token 消耗上限和 API 調用頻率限制
  • 防止 agent 在遞迴迴圈中產生天價雲端帳單

2.6 MCP 協議（Model Context Protocol）深入分析

📖 Anthropic — Model Context Protocol

什麼是 MCP

  • Anthropic 提出的開放協議，讓 AI 模型能以標準化方式連接外部工具和資料源
  • 類似於「AI 的 USB-C」——一個統一接口標準

2026 年的進展

  • 採用率快速增長：主流 IDE（Cursor、Windsurf）、開發工具鏈相繼支持
  • Server 生態系：從最初的幾十個 MCP server 增長到數百個，覆蓋資料庫、API、搜尋引擎、代碼分析等
  • 多模型支援：不只 Claude，OpenAI 和 Google 模型也開始支持 MCP 連接

對企業的意義

  • MCP 讓企業能以標準化方式連接內部系統（ERP、CRM、HR）到 AI agent
  • 不需要為每個模型和每個系統寫客製化整合
  • 風險：標準化接口也意味著標準化攻擊面——prompt injection 可以透過 MCP 傳播

3.1 企業採購：Agentic AI 的第一個高價值場景

📖 Deloitte — Autonomous Intelligence & Agentic Procurement

場景描述

一個 agentic 應用持續交叉比對供應鏈庫存和即時供應商價格（在 ERP 系統中），然後在預設財務參數內獨立授權採購訂單，只有在出現偏差時才暫停請求人類批准。

為什麼這是第一個高價值場景

  • 採購是成本密集流程：企業 40-70% 的支出在採購
  • 決策瓶頸明確：價格比對、庫存檢查、供應商選擇——都是可規則化的決策
  • 人類在迴圈中的成本高：每一個採購訂單的審批週期平均 2-5 天
  • Agentic AI 可以將週期壓縮到分鐘級

部署前提條件（Deloitte 框架）

  • Agent 必須在 ERP 中有可驗證的身份
  • 讀取的價格數據必須足夠即時以具合約效力
  • 操作在法律和合規部門正式認可的審批閾值內
  • 任何一個依賴項未解決，自主執行的案例就會完全崩潰

概念

AI Agent 的能力邊界急速擴張，讓「一個人 + AI 團隊」的運作模式成為可能。過去需要一整個部門完成的工作——行銷、客服、財務分析、程式開發——現在可以由一個「超級個體」搭配 AI Agent 群完成。

關鍵論點

  • Anthropic 引發的 SaaS 死亡辯論：Anthropic 曾暗示 AI Agent 將取代許多 SaaS 平台的功能，引發軟體股下跌。核心邏輯是：如果 AI Agent 可以直接操作 API、串接流程，傳統 SaaS 的中介角色就不再必要。這意味著原本需要採購多個 SaaS 工具的團隊，現在可能一個人就能搞定。
  • 「Build, Buy, Borrow, or Bot」人才策略（AI Fieldbook, 2026/4）：企業面對人才缺口時，第四個選項是「Bot」——用 AI Agent 補充人力。這直接改變了組織規模的計算公式。
  • 一人公司 != 自由工作者：傳統自由工作者是「什麼都自己來」，AI 時代的一人公司是「自己當指揮家，AI Agent 當樂團」。差別在於產出規模可以跟一個 20 人團隊相當。

5.5 變革管理：如何讓組織擁抱 AI

📖 AI Fieldbook — Overcoming resistance: Getting your organization to embrace AI

抵抗的四種型態（AI Fieldbook, 2026/4/29）

抵抗類型表現根本原因

  • 透明領導：不隱瞞 AI 的影響，不過度吹捧好處；如果 AI 可能導致角色變動，直說，並說明支援方案
  • 正面處理工作恐懼：不要講空話（「AI 不會取代你，它會增強你」），要具體說明哪些工作會受影響、怎麼受影響、有什麼支援

  • 讓早期勝利可見：快速、可見的成功建立信任；廣泛分享成功故事，把功勞歸給參與的人
  • 建立安全實驗空間：容許失敗的環境，從失敗中學習跟從成功中學習一樣重要

AI Expo 2026 的開發者工作流座談會（Valae, Charles River Labs, Knight Frank）指出：AI Copilot 加速了代碼生成，但同時迫使開發者把更多精力放在 review 和架構上。這是一個根本性的角色轉變。

Agent Mode 的五條軍規

  • 小任務、小提交：不要一次讓 Agent 改 20 個檔案。一次一個功能點，提交後再下一個
  • 先寫測試，再讓 Agent 實作：TDD + AI = 最安全的組合。紅燈 → 讓 Agent 實作 → 綠燈
  • 每次產出都要 review：Agent 的代碼可能「看起來對但邏輯有問題」。特別注意：
- 邊界條件是否處理

- 錯誤處理是否完整

- 是否引入了不必要的依賴

  • 用 .cursorrules 定義規範：讓 Agent 遵守團隊的 coding style、命名慣例、架構模式
  • MCP 是你的超能力：連接內部 API 文件、資料庫 schema、設計系統，讓 Agent 在你的上下文裡工作

什麼是 MoE？

Mixture of Experts（MoE）是一種稀疏神經網路架構。傳統 Dense 模型對每個 token 都啟動全部參數，但 MoE 將前饋層（FFN）替換為多個「專家」+ 一個路由器（Router）。每個 token 只啟動其中 K 個專家，大幅降低計算成本。

傳統 Dense：  Token → [全部 70B 參數] → 輸出（每次都用 70B）
MoE：         Token → Router → 選 2/8 專家 → 加權合併 → 輸出（每次只用 ~13B）

MoE 的核心權衡

特性Dense 模型MoE 模型
計算成本高（每 token 啟動全部參數）低（只啟動一小部分）
VRAM 需求較低較高（需載入所有專家權重）
訓練複雜度標準高（需負載平衡防專家崩潰）
推理速度慢（大模型）快（等同小模型）
通訊開銷低高（多 GPU 間的專家並行）

MoE 三個時代的演進

1️⃣ Mixtral 時代（2023–2024）

  • Mixtral 8x7B（2023/12）：首次確立開源 MoE 典範，8 個專家 + Top-2 路由
  • Mixtral 8x22B（2024/4）：141B 總參數 / 39B 啟動
  • 特徵：少量大專家、Top-K 路由、輔助負載平衡損失

2️⃣ 細粒度 MoE 時代（2025–2026）

  • DeepSeek V3（2024/12）：256 個路由專家 + 1 共享專家 + Top-8 路由 🔥
  • Qwen 3：128 專家 Top-2（30B-A3B 版本僅 3B 啟動）
  • 特徵：更多更小的專家、共享專家捕獲通用知識、路由專家專精化
  • 結果：DeepSeek V3（671B/37B）大幅超越 Mixtral 8x22B（141B/39B），在相似推理成本下品質更高

3️⃣ DeepSeek Sparse Attention 時代（2026）

  • DeepSeek V3.2 / V4：將 MoE 概念延伸到注意力層
  • DSA（DeepSeek Sparse Attention）：每個 query token 只注意相關的 key token 子集
  • 這讓 DeepSeek V4 能在生產環境支援 1M token context——原生 dense attention 在 1M tokens 下的計算和 KV-cache 記憶體成本是天文數字

MoE 為什麼在 2026 成為主流？

  • 推理經濟性：前沿模型超過 70B dense 參數後，純 dense 推理成本變得不可接受。MoE 讓 1T 模型以 32B 的成本運行
  • 更好的專業化：細粒度專家能更精確地專精——數學 token → 數學專家，程式碼 token → 程式碼專家
  • 共享專家突破：DeepSeek 的共享專家設計讓通用知識不必在每個路由專家中重複，大幅提升效率
  • DSA 延伸：從 FFN 層延伸到 Attention 層，解鎖了 1M+ context 的生產可行性

MoE 的部署挑戰

  • VRAM 是最大瓶頸：1T 模型即使只啟動 32B，仍需載入全部 1T 權重。需要 H100/H200 等大 VRAM GPU
  • 專家崩潰（Expert Collapse）：路由器可能偷懶只送 token 到 1-2 個專家，訓練時必須加負載平衡損失
  • 專家並行通訊：多 GPU 叢集中，token 需傳送到持有目標專家的 GPU，對 NVLink 頻寬要求極高
  • 量化是出路：GGUF / EXL2 格式能讓 MoE 模型壓進消費級 GPU

7.8 2026 年 5 月排行榜趨勢總結

五大趨勢

  • 閉源編程差距仍然巨大：SWE-Bench 上閉源領先 21.8 分，這是目前開源最大的能力鴻溝。Claude Opus 4.7 (87.6%) 遠超 Kimi K2 Instruct (65.8%)
  • 數學和推理差距幾乎消失：AIME 2025 和 HLE 上，開源與閉源差距僅 0.9 分。DeepSeek-R1 在 MATH-500 上甚至超越閉源
  • 小模型正在顛覆大模型：Gemma 4 31B 用 30.7B 參數打出了 GPQA 84.3——超越 671B 的 DeepSeek-R1 (71.5) 和 1T 的 Kimi K2 (75.1)
  • MoE 架構成為開源主流：DeepSeek-R1 (671B MoE)、Kimi K2 (1T MoE)、Llama 4 (400B MoE)——用更少的啟動參數實現更大的總能力
  • Context Window 軍備競賽：Gemini 3 Pro 和 Llama 4 都達到 10M tokens，GPT-5.5 達 1M。長 context 已從賣點變成基本要求
