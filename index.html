第一段：AI 造成的進步與組織／工作轉變

1.1 企業從「輔助智慧」走向「自主智慧」的三階段演化

Deloitte AI & Insights 負責人 Prakul Sharma 提出了企業智慧成熟度曲線的三個階段，這是目前業界最清晰的框架之一：

階段名稱特徵代表應用
第一階段Assisted Intelligence（輔助智慧）AI 和分析幫人解讀資訊，人做最終決策傳統 BI 儀表板、推薦系統
第二階段Augmented Intelligence（增強智慧）機器學習輔助人類決策，人仍在迴圈中GenAI 聊天機器人、Copilot、摘要工具
第三階段Autonomous Intelligence（自主智慧）AI 在設定邊界內自行決策並執行，人設護欄但不驅動每一步Agentic AI 自動採購、自主交易系統
關鍵洞察：目前大多數企業的 GenAI 應用（聊天機器人、文件摘要）停留在第二階段，但真正的經濟價值在第三階段。GenAI 產生一個答案，而自主智慧追求一個結果——它會對目標進行推理、調用工具和資料、根據條件變化調整行動。

「個人副駕駛」的規模化陷阱

TechEx North America 2026 多場會議反覆提及一個現象：「個人 Copilot」陷阱。一個 AI 工具在單一員工桌上表現很好，C-suite 主管自己用了覺得效率提升，全公司都很興奮——但要把這個成功經驗從一個人擴展到整個部門，甚至整個公司，卻處處碰壁。原因是：

  • 個人效率 ≠ 組織效率：單人省下的時間，不等於流程被重構
  • Pilot 用精選資料和冠軍團隊手動跑，掩蓋了缺失的身份控制、過期資料、延遲的合規審查
  • 一旦進入正式部署，這些「治理負債（governance debt）」立刻成為結構性阻礙

1.2 OpenAI 新加坡實驗室與 IMDA Agentic AI 框架

OpenAI 在新加坡設立新區域實驗室，投資 S$3 億（約 US$2.25 億），將招聘 200+ 員工，涵蓋工程、研究、政策等職位。這不只是擴張——它伴隨一個更重要的治理框架：

IMDA 的 Agentic AI 風險分級框架

新加坡資通訊媒體發展局（IMDA）與 OpenAI 合作推出全球首個針對 Agentic AI 的風險分級治理框架，分為四個風險等級：

風險等級定義治理要求
Tier 1 — 極低風險資訊檢索、簡單問答基本透明度要求
Tier 2 — 低風險內部流程自動化、非關鍵任務人類監督 + 活動日誌
Tier 3 — 中風險涉及外部系統互動、財務操作人類在迴圈 + 審批閾值 + 完整審計軌跡
Tier 4 — 高風險自主執行交易、關鍵基礎設施操作即時監控 + 隨時撤銷權限 + 監管機構通報
為什麼重要：這是全球第一個不是「先發展再管制」，而是「隨風險遞增要求遞增治理」的框架。它解決了企業最頭痛的問題——不是不部署 Agentic AI，而是不知道部署到什麼程度需要什麼治理。

1.3 美國政策：川普廢除 AI 行政命令的完整脈絡

川普政府正式廢除了拜登時期的 AI 行政命令（EO 14110），該命令要求大型 AI 模型開發者向政府報告訓練細節和安全測試結果。廢除的背景：

  • Musk 和 Zuckerberg 積極遊說：認為報告要求增加合規成本、削弱美國 AI 公司競爭力
  • 核心論述：「對中國競爭力」——如果美國 AI 公司被監管束縛，中國會超越
  • 替代方案：川普政府傾向於由市場自行調節，而非預先監管
  • 實際影響：目前沒有聯邦層面的強制性 AI 安全報告要求，各州自行立法（如加州 SB 1047 的各種變體仍在討論中）

Nvidia H200 中國出口的結構性僵局

一個更具戲劇性的政策故事：川普帶 Jensen Huang 去北京參加川習會，會後對記者說「something could happen」——但什麼都沒發生。

僵局的真實結構：
  • 美國已授權：約 10 家中國公司（阿里巴巴、騰訊、字節跳動、京東等）獲得最多 75,000 顆 H200 的出口許可，聯想和富士康為授權經銷商
  • 中國不收貨：北京指示中國科技公司將 Nvidia 晶片限用於海外業務，同時支持國產製造
  • 兩個要求互相矛盾：美國要求 H200 只能在中國境內使用；北京要求中國公司只用國產晶片做國內業務
  • 華為趁虛而入：DeepSeek V4 已適配華為 Ascend 晶片進行訓練（不只是推理），這是首次中國前沿模型在訓練層面跑國產晶片
  • Nvidia 中國營收已從 20%+ 跌至約 5%，本季度指引假設中國營收為零
這意味什麼：中國第二大 AI 市場的硬體架構正在被政府指令而非技術基準決定。這不是貿易姿態，是結構性賭注——賭華為效能差距會縮小得夠快。

1.4 歐盟 AI Act 的 Agentic AI 治理挑戰

歐盟 AI Act 在 2026 年進入執行階段，但 Agentic AI 帶來了立法者未預見的挑戰：

Article 9：持續風險管理

高風險 AI 的風險管理必須是持續的、基於證據的過程，內建於部署的每個階段（開發、準備、生產），並持續審查。

Article 13：可解釋性要求

高風險 AI 系統必須設計成讓部署者能理解其輸出。這意味著：

  • 第三方 AI 系統不能是不透明的程式碼塊
  • 必須提供足夠文件確保安全合法使用
  • 模型選擇和部署方式同時是技術和法規考量

Agentic AI 特有的治理工具

  • Asqav Python SDK：用密碼學簽名每個 agent 的操作，並將所有記錄連結到不可變的 hash chain（類似區塊鏈技術）。如果有人修改或刪除記錄，chain 驗證就會失敗
  • Agentic Asset List（代理資產清單）：企業必須維護所有運行中 agent 的登記冊，每個 agent 需唯一識別，記錄其能力和授予的權限
  • 即時撤銷機制：在高風險場景中，必須能隨時撤銷 agent 的權限

1.5 微軟開源 Agent 治理工具包

微軟發布了開源的 Agent Governance Toolkit（[GitHub](https://github.com/microsoft/agent-governance-toolkit)），針對 Agentic AI 的運行時安全，這是目前最實用的企業治理工具：

核心設計：攔截工具調用層

框架在語言模型和企業網路之間放置一個策略執行引擎：

  • 每次 agent 嘗試調用外部工具，toolkit 攔截請求並對照治理規則檢查
  • 如果行為違反策略（例如：只授權讀取庫存數據的 agent 嘗試發送採購訂單），阻擋 API 調用並記錄事件
  • 安全團隊獲得可驗證、可審計的每一個自主決策軌跡
  • 開發者不需要在每個模型的 prompt 中硬編碼安全協議——安全策略與應用邏輯完全解耦

為什麼開源？

  • 如果微軟把這個功能鎖在自家平台，開發團隊會為了趕 deadline 直接繞過它
  • 開源讓安全控制能融入任何技術棧——無論是本地開源模型、Anthropic、還是混合架構
  • 開放標準讓整個資安生態系能在此基礎上建立商業儀表板和事件響應整合

成本控制層面

  • Agentic 系統在推理和執行的連續迴圈中燃燒 API token
  • Toolkit 可設定 token 消耗上限和 API 調用頻率限制
  • 防止 agent 在遞迴迴圈中產生天價雲端帳單

1.8 企業 AI 部署的「決策審計」方法論

Deloitte 的 Sharma 提出具體的操作方法：

  • 決策審計：選擇 1-2 個價值鏈，其結果被「決策」而非「任務」瓶頸
  • 映射決策流程：誰有資料？誰有權限？交接在哪裡斷裂？需要什麼行動？哪裡需要判斷？
  • 暴露資料和治理缺口：這些問題會同時揭示自主智慧能創造經濟價值的地方，以及可能導致 pilot 失敗的缺口
  • 建立基礎層：AI/Agentic Fabric、資料、評估、agent 身份、human-in-the-loop 模式
  • 在第一個價值鏈上證明有效，然後作為模板規模化

「決策級資料」vs「報告級資料」的關鍵區別

報告級資料決策級資料
每夜/每週批次處理即時或近即時
為儀表板消費而結構化帶有時間戳記、可追溯來源、存取控制
供人類判斷後行動供 agent 直接行動
去除了值如何衍生的譜系保留完整譜系（provenance）
風險案例：如果 agent 根據過時的批次處理數據執行交易，可能使用過期的價格層級或過時的合規框架——這是極端風險。

1.9 Shadow AI：安全團隊跟不上部署速度

TechEx Cyber Security 軌道提出的「速度差距（velocity gap）」：

  • AI 部署成功時，獲得吸引力的速度極快
  • 但業務部門採用 GenAI 的速度超過安全團隊治理和確保企業安全的能力
  • Shadow AI：員工將敏感資料放入未批准的 AI 工具，或已批准的 AI 系統缺乏邊界和管理
  • 攻擊面在安全團隊不知情的情況下擴大
  • 零信任作為答案：人類和機器都需要身份和權限證明；自動化工作流程必須服從與 IT 堆疊中其他元素相同的權限模型

1.10 JBS Dev 的實務觀點：資料不必完美才能開始

JBS Dev 總裁 Joe Rose 的核心論點：

  • 「資料必須完美才能做 AI」是迷思——供應商和顧問推動這個說法是為了賣資料湖和多年轉型專案
  • 工具前所未有地好：LLM 可以理解半成品 prompt
  • 醫療行業案例：客戶遷移計費系統，記錄混亂（PDF、圖片混雜；醫生名寫在病人名欄位等），GenAI 能從簡單 prompt 中提取乾淨數據，從 OCR 到圖片轉文字，後續用 agentic 方法比較客戶記錄和保險合約的計費費率
  • 漸進式自動化：從 20% 自動化開始，然後 40%、60%、80%，逐步增長
  • 未來焦點轉向成本和可攜性：模型能力的跳躍會放緩，重點轉向「如何讓成本可持續」和「如何讓模型跑在筆電或手機上而非資料中心」
  • 不要買 SaaS——自己建：幾乎每家公司都有雲端存在，三大雲端的工具足以明天就開始實施 agentic 工作負載，不需要新的軟體授權和培訓

2.1 GPT-5.5：OpenAI 第一個為 Agentic 工作重訓的基礎模型

發布背景

  • 發布日期：2026 年 4 月 23 日
  • 定位：OpenAI 第一個從零重訓（retrained from scratch）的基礎模型，專門為 agentic 工作設計

核心能力

能力說明
規劃（Planning）能夠分解複雜任務為多步驟計畫，動態調整執行順序
工具使用（Tool Use）原生支持多工具調用，不再需要複雜的 prompt 工程來觸發
自我檢查（Self-checking）在執行過程中自行驗證中間結果，降低錯誤傳播
獨立任務（Independent Tasks）能在最少人類介入下完成端到端任務

定價

  • API 成本為前代的 2 倍，但 agentic 工作流整體效率提升意味著單位任務成本可能下降
  • 反映出「模型能力已趨近商品化，差異化在 agentic 能力」的市場趨勢

產業意義

  • 這不是一個更大的 GPT-5——它是一個為 agent 而非 chat 設計的模型
  • 標誌著基礎模型從「通用問答」向「任務執行引擎」的範式轉移
  • 其他廠商必然跟進，agentic-native 模型將成為新基準

2.4 阿里巴巴 Zhenwu M890：Agent 優化晶片

核心規格

  • 效能為前代的 3 倍
  • 專門為 agentic 工作負載優化——不是通用推理，是 agent 的多步驟推理和工具調用
  • 阿里巴巴發布了多年矽路線圖，表明這不是一次性產品

意義

  • 第一個明確標榜「agent 優化」的 AI 晶片
  • 意味著晶片設計正在適應 agentic AI 的獨特需求：多輪推理、工具調用、狀態管理
  • 可能帶動整個晶片行業從「更大更快的推理」向「更聰明的 agent 執行」轉向

2.5 Anthropic Project Glasswing：AI 找漏洞的里程碑

計畫概述

  • 50+ 合作夥伴的協作計畫，目標是保護關鍵軟體
  • 使用 Claude Mythos Preview 模型

成果

  • 發現 10,000+ 高/嚴重級別漏洞
  • 進度現在受限於漏洞修復的速度，而非發現的速度
  • 這是 AI 安全研究的一個轉折點：瓶頸從「找問題」變成了「修問題」

方法論

  • Claude 被用來對開源軟體和關鍵基礎設施代碼進行系統性審計
  • 不同於傳統 fuzzing 和靜態分析，AI 模型能理解代碼語義和上下文
  • 發現的漏洞類型包括：記憶體安全問題、邏輯錯誤、競態條件等

2.6 MCP 協議（Model Context Protocol）深入分析

什麼是 MCP

  • Anthropic 提出的開放協議，讓 AI 模型能以標準化方式連接外部工具和資料源
  • 類似於「AI 的 USB-C」——一個統一接口標準

2026 年的進展

  • 採用率快速增長：主流 IDE（Cursor、Windsurf）、開發工具鏈相繼支持
  • Server 生態系：從最初的幾十個 MCP server 增長到數百個，覆蓋資料庫、API、搜尋引擎、代碼分析等
  • 多模型支援：不只 Claude，OpenAI 和 Google 模型也開始支持 MCP 連接

對企業的意義

  • MCP 讓企業能以標準化方式連接內部系統（ERP、CRM、HR）到 AI agent
  • 不需要為每個模型和每個系統寫客製化整合
  • 風險：標準化接口也意味著標準化攻擊面——prompt injection 可以透過 MCP 傳播

3.1 企業採購：Agentic AI 的第一個高價值場景

場景描述

一個 agentic 應用持續交叉比對供應鏈庫存和即時供應商價格（在 ERP 系統中），然後在預設財務參數內獨立授權採購訂單，只有在出現偏差時才暫停請求人類批准。

為什麼這是第一個高價值場景

  • 採購是成本密集流程：企業 40-70% 的支出在採購
  • 決策瓶頸明確：價格比對、庫存檢查、供應商選擇——都是可規則化的決策
  • 人類在迴圈中的成本高：每一個採購訂單的審批週期平均 2-5 天
  • Agentic AI 可以將週期壓縮到分鐘級

部署前提條件（Deloitte 框架）

  • Agent 必須在 ERP 中有可驗證的身份
  • 讀取的價格數據必須足夠即時以具合約效力
  • 操作在法律和合規部門正式認可的審批閾值內
  • 任何一個依賴項未解決，自主執行的案例就會完全崩潰

3.2 醫療計費對帳：從混亂數據到結構化

JBS Dev 案例

  • 客戶：醫療行業客戶，遷移計費對帳系統
  • 數據狀態：極度混亂——PDF 和圖片混雜；醫生名寫在病人名欄位；程序代碼不規範
  • GenAI 層：從簡單 prompt 提取乾淨數據，OCR 圖片、提取 PDF 文字
  • Agentic 層：比較客戶記錄和保險合約，驗證計費費率是否正確
  • 漸進自動化：20% → 40% → 60% → 80%，逐步擴展

為什麼這個案例重要

  • 證明了「數據不必完美也能開始」——LLM 可以理解半結構化甚至非結構化數據
  • 展示了 GenAI → Agentic AI 的漸進路徑：先解決數據理解，再解決決策自動化
  • 醫療計費是典型的高重複性、高錯誤率流程——AI 的理想目標

4.5 Deloitte 決策審計法：從 Pilot 到生產的正確路徑

步驟

1. 選擇價值鏈
  • 不要從技術出發，從「哪裡的結果被決策（而非任務）瓶頸」出發
  • 選 1-2 個價值鏈開始
2. 映射決策流程
  • 誰有資料？
  • 誰有權限？
  • 交接在哪裡斷裂？
  • 需要什麼行動？
  • 哪裡需要判斷？
3. 暴露缺口
  • 這些問題會同時揭示：自主智慧能創造價值的地方 + pilot 可能失敗的缺口
4. 建立基礎層
  • AI/Agentic Fabric
  • 決策級資料（不是報告級！）
  • 評估機制
  • Agent 身份管理
  • Human-in-the-loop 模式
5. 在第一個價值鏈上證明有效，然後作為模板規模化

關鍵原則

  • Pilot 不是實驗，是第一個生產實例——帶同樣的 eval、身份模型和治理
  • 治理負債（Governance Debt）：Pilot 中跳過的控制、審計軌跡、風險框架，會在正式部署時成為阻礙
  • 模型不是瓶頸：前沿能力已趨近商品化，瓶頸在上游——工作流映射和資料品質

4.6 解決「Pilot Purgatory」：5 個實用建議

TechEx 2026 多場會議的共識：

  • 不要自動化壞流程：先修復工作流，再讓 agent 介入。否則你只是把壞流程跑得更快
  • 決策級資料優先：確保 agent 讀取的數據帶有時間戳、來源譜系和存取控制，而不是夜間批次的摘要
  • 治理從第一天開始：不要等 pilot 成功再加安全——把 eval、身份、合規當作 first-class requirement
  • 漸進式自動化：從 20% 開始，逐步到 40%、60%、80%。不要一次跳到全自動
  • 監控 token 成本：Agentic 工作流的多輪推理會快速累積 API 成本，設定消耗上限

4.7 Shadow AI 防治：零信任 + Agent 治理

問題

  • 員工使用未批准的 AI 工具處理敏感資料
  • 已批准的 AI 系統缺乏邊界管理
  • 攻擊面在安全團隊不知情的情況下擴大

解決方案

措施實作方式
零信任人類和機器都需要身份和權限證明，預設拒絕
Agent 登記冊所有運行中的 agent 必須在登記冊中，記錄能力和權限
運行時攔截Microsoft Agent Governance Toolkit 攔截所有工具調用
數據治理 = 資安治理兩者已不可分——資料外洩和 AI 幻覺是同一個問題的兩面

Section 5：組織轉變 — AI 時代的組織形態與實例

時間範圍：2026 年 5 月第三週及近期重要討論
資料來源：AI Fieldbook、TechEx North America 2026、Deloitte、Laserfiche、Anthropic 相關討論

5.1 一人公司與「藝人專案」模式

概念

AI Agent 的能力邊界急速擴張，讓「一個人 + AI 團隊」的運作模式成為可能。過去需要一整個部門完成的工作——行銷、客服、財務分析、程式開發——現在可以由一個「超級個體」搭配 AI Agent 群完成。

關鍵論點

  • Anthropic 引發的 SaaS 死亡辯論：Anthropic 曾暗示 AI Agent 將取代許多 SaaS 平台的功能，引發軟體股下跌。核心邏輯是：如果 AI Agent 可以直接操作 API、串接流程，傳統 SaaS 的中介角色就不再必要。這意味著原本需要採購多個 SaaS 工具的團隊，現在可能一個人就能搞定。
  • 「Build, Buy, Borrow, or Bot」人才策略（AI Fieldbook, 2026/4）：企業面對人才缺口時，第四個選項是「Bot」——用 AI Agent 補充人力。這直接改變了組織規模的計算公式。
  • 一人公司 != 自由工作者：傳統自由工作者是「什麼都自己來」，AI 時代的一人公司是「自己當指揮家，AI Agent 當樂團」。差別在於產出規模可以跟一個 20 人團隊相當。

實例觀察

模式說明來源
藝人專案（Solo Project）單人靠 AI Agent 執行完整產品開發週期，從 ideation → design → coding → deployment社群討論趨勢
AI Agent 群組運作一人指揮多個 Agent 分工：研究 Agent 負責資料收集、寫作 Agent 負責內容、審計 Agent 負責品質把關Laserfiche AI Agents（2026/5 發布）
SaaS 替代模式用 AI Agent 直接串 API 取代購買 CRM/ERP/HR 軟體Anthropic SaaS 辯論

5.2 組織扁平化與敏捷化

AI 壓平組織圖的趨勢

AI 讓中層管理者的「資訊轉運」角色被取代。過去中層主管的功能是：向上彙報、向下傳達、橫向協調。當 AI Dashboard 可以即時呈現各團隊狀態、AI Agent 可以自動協調跨部門流程時，中層的管理必要性下降。

AI Fieldbook 領導力框架（2026/5/14）

AI Fieldbook 提出「AI-ready 組織四大支柱」，明確指出 AI 準備度 = 30% 技術 + 70% 組織：

支柱核心要求具體內容
🎯 StrategyAI 策略必須連結商業策略不是「找問題配解方」，而是從業務痛點出發；用商業語言定義成功指標，不是技術指標
📊 Data數據是 AI 的燃料關鍵數據資產需識別、文檔化；數據品質需量測與改善；數據治理需有明確政策
👥 Talent人才策略涵蓋 Build/Buy/Borrow/Bot不只聘 data scientist，要讓每個部門都有能力參與 AI 轉型；投資全員 upskilling
🧬 Culture文化是最難但最重要的一環領導者要示範數據驅動決策；鼓勵實驗，容許失敗；打破「不是這裡發明」症候群

新領導力堆疊（AI Fieldbook, 2026/4/6）

未來的組織領導者需要一套新的能力組合：

  • AI 流暢度（不是技術專家，但要能跟技術團隊有效對話）
  • 問對問題的能力 > 有答案的能力
  • 變革領導力：組織轉型最重要的技能
  • 在不確定中做決策：AI 模型會犯錯、法規在變、最佳實踐還在形成中
  • 持續學習心態：承認不知道、持續更新理解

5.3 從 Pilot 到規模化：組織調整的現實挑戰

70% 的 AI Pilot 無法規模化

AI Fieldbook（2026/5/19）的研究指出：高達 70% 的 AI 試點專案從未進入生產環境。

Pilot Trap 的四階段

  • 興奮期：找到有潛力的 use case，組小團隊，分配資源
  • 技術成功期：原型跑通了，模型準確率高，demo 很好看
  • 現實衝擊期：要上線時才發現——數據品質在規模下崩潰、系統整合超複雜、沒人願意負責、用戶抗拒
  • 停滯期：不殺也不推，資源被抽走，冠軍離開

五大失敗原因與組織對策

失敗原因組織該怎麼調整
指標只看技術（accuracy/precision），不看商業價值從第一天就用商業語言定義成功：什麼決策會不同？什麼流程會改善？
Pilot 用的是「乾淨數據」，上線遇到「髒數據」就崩潰Pilot 階段就使用生產環境的真實數據，提早暴露問題
低估系統整合難度IT 和維運團隊從第一天就參與，不要等到要部署才叫他們
組織沒準備好，用戶不信任/不會用Change management 從 Pilot 階段就開始，不是上線後才做
沒人負責從 Pilot 推到生產指定明確的 owner，不只是「建」，還要負責「推」

5.4 Fortune 500 製造業規模化實例：40 工廠的 AI 部署

案例背景（AI Fieldbook, 2026/4/22）

一家 Fortune 500 製造商（化名 GlobalMfg）在 3 年內將 AI 部署到 40 座工廠，橫跨三大洲，創造超過 $2 億美元 的已驗證價值。

關鍵做法

階段做了什麼組織調整
Pilot（6 個月）只做一條生產線的預測性維護每個步驟都文檔化，量測的是商業成果不是技術指標，一線員工全程參與
選擇 Rollout 策略放棄「大爆炸」（風險太高）和「有機生長」（太慢），選擇「分波次」每 5-8 座工廠一波，18 個月完成
標準化所有工廠用同一套方案、同一套做法局部差異只在「有明確商業理由」時才允許
廠長當冠軍廠長是成敗關鍵投資大量心力讓廠長成為擁護者，讓他們在成功時成為英雄
中央支援團隊15 人核心團隊（data scientist + engineer + change management）工廠不用自己搞，中央提供專業支援
CEO 可見度CEO 每月 review、親自到工廠信號明確：AI 轉型是優先事項，不是 side project

踩過的坑

  • 低估數據差異：不同工廠的設備世代不同，sensor 配置不同，數據 harmonization 比預期複雜得多
  • 前期 Change management 不足：早期波次太偏技術，訓練和採用做得太少；後期加倍投入，效果好很多
  • Pilot fever：第一個 use case 成功後，大家急著同時啟動多個 AI 專案，結果分散資源，全部變慢

最終成果

「We're an AI company now. Every improvement conversation starts with, how can AI help?」 — VP of Operations

5.5 變革管理：如何讓組織擁抱 AI

抵抗的四種型態（AI Fieldbook, 2026/4/29）

抵抗類型表現根本原因
工作被取代的恐懼「AI 會搶我的飯碗」生存焦慮，這是最真實的恐懼
專業自主權的喪失「我花了 10 年學的東西，AI 一秒就做完了」身份認同被挑戰
不信任技術「我們試過類似的東西，都失敗了」過去創傷
實際擔憂「這真的會讓我工作更輕鬆，還是更多事？」合理的質疑
組織層面質疑「誰受益？改善工作還是砍成本？」信任問題

六個有效策略

  • 透明領導：不隱瞞 AI 的影響，不過度吹捧好處；如果 AI 可能導致角色變動，直說，並說明支援方案
  • 正面處理工作恐懼：不要講空話（「AI 不會取代你，它會增強你」），要具體說明哪些工作會受影響、怎麼受影響、有什麼支援
  • 讓人參與過程：讓 end user 參與問題識別、方案設計、結果評估；參與創造的人遠比被施加方案的人更容易擁抱改變
  • 從願意採用的人開始：不要先說服懷疑者，找 early adopters 示範成功，創造正向同儕壓力
  • 讓早期勝利可見：快速、可見的成功建立信任；廣泛分享成功故事，把功勞歸給參與的人
  • 建立安全實驗空間：容許失敗的環境，從失敗中學習跟從成功中學習一樣重要

5.6 AI 轉型的五階段 Playbook

AI Fieldbook 轉型框架（2026/5/18）

階段關鍵任務組織調整重點
Phase 1：願景與策略為什麼做 AI？成功長什麼樣子？由上而下建立清晰願景，與商業策略掛鉤
Phase 2：評估與規劃現在在哪？要往哪去？評估數據、技術、人才、文化現狀；找 gap；按價值和可行性排優先序
Phase 3：基礎建設數據底座 + 技術能力 + 治理框架開始文化變革；不是先買工具，而是先建基礎
Phase 4：執行與規模化執行優先 use case，放大成功慶祝勝利、從失敗學習、規模化成功做法
Phase 5：制度化AI 成為組織運作的一部分AI 嵌入業務流程；發展 AI 原生人才；治理隨成熟度演進

成功要素

  • ✅ 高層持續承諾（不是 launch 完就消失）
  • ✅ 明確問責（有人端到端負責整個轉型）
  • ✅ 足夠資源（人、技術、變革管理都要投資）
  • ✅ 耐心與堅持（轉型是以年計的旅程，不是以月計的專案）

  • 應付帳款：找出逾期發票並導向正確團隊處理

Bot = 用 AI Agent 填補人力缺口

這不是取代人，是重新定義「一個人能管多大的範圍」。

5.9 給組織領導者的行動清單

  • 🔍 盤點現有 Pilot：哪些在 limbo？用 5 大失敗原因對照，找出卡住的原因
  • 👤 指定 Owner：每個 AI 專案都要有人負責從 Pilot 推到生產
  • 📏 改指標：把技術指標（accuracy, latency）換成商業指標（revenue impact, cost saved, time saved）
  • 🧪 真實數據測試：Pilot 階段就用生產環境的髒數據，不要等到上線才發現問題
  • 🏭 從一個廠/一個團隊開始：GlobalMfg 的模式——小範圍成功 → 標準化 → 分波次推廣
  • 🎤 說故事：每一個成功案例都要廣為宣傳，讓 early adopters 成為英雄
  • 🤖 盤點「Bot 可補」的缺口：哪些人力缺口可以用 AI Agent 填？列出清單，評估可行性
  • 🛡 建治理框架：在部署 Agent 之前就建立治理，不是出事後才補
  • 📈 量測採用率：不只追蹤部署數量，要追蹤「有多少人真的在用」
  • ⏳ 給自己時間：AI 轉型是 3 年的旅程，不是 3 個月的專案
